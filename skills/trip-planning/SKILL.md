# 行程规划技能

本 skill 负责把用户的出行目标整理成可执行安排。地点、路线、天气等事实查询使用 `location-services` 或 `travel-resources`；机票、酒店和房型等资源查询使用 `travel-resources`；明确要创建或修改日历时使用 `schedule-management`。

## 协作 skill

- 需要地点、路线、天气、机票、机场代码、酒店、房型或其他出行事实时，用 `travel-resources` 统筹补齐；不要创建订单。
- 只有纯地图/天气任务且不需要机酒或报告时，可直接用 `location-services`。
- 用户明确要提醒、安排到日历、创建或更新日程时，用 `schedule-management` 落地时间边界。
- 用户要持续留意票务开售、机酒降价、展览/活动更新、周末候选活动，或定期帮他找值得去的地方时，不创建隐藏跟踪；只能落具体日程、继续在聊天里规划，或说明当前不支持持续跟踪。
- 用户要攻略、报告、可分享行程或文件化输出时，用 `ling-report` 生成 HTML。
- 旅行、差旅、多日行程、机酒组合、多人计划或信息较多的行程规划，默认用 `ling-report` 生成 HTML；聊天里只给少量摘要和文件链接。

## 适用范围

使用本 skill 处理：
- 多日旅游、周末游、一日游、城市散步。
- 出差、探店、约会、看展、接送、拜访、办事。
- “帮我安排怎么走”“今天出去玩什么”“从 A 到 B 顺路做几件事”。
- 根据天气、时间、距离、营业信息或用户偏好调整行程。

## 规划原则

- 先明确约束：日期、出发地、目的地、人数、交通方式、预算、体力、偏好、必须去/不能去。
- 信息足够时直接给出方案；缺关键字段、候选冲突或高风险误判时再问。
- 不要把用户没给出的约束当事实。出发地、返程目的地、交通方式、住宿区域、预算、同行人、体力节奏和兴趣偏好都不能凭空补全。
- 已知信息足以给“待确认草案”时，可以用明确标注的假设继续，例如“我先按从你当前城市出发、公共交通优先做草案”；这些假设必须写在方案前，并把需要用户确认的问题收束到 1-3 个。
- 如果缺失信息会显著改变路线、预算或可预订资源，先问关键问题，不生成看似确定的完整行程。
- 优先减少折返，按地理邻近、营业时间、交通耗时和用户精力排序。
- 普通外出重视少打扰和可执行；旅游规划重视节奏、体验和备选。
- 不编造营业时间、票价、交通耗时、天气或地点状态。
- 不编造机票价格、酒店价格、库存、房型、早餐、退改规则或可售状态；查不到就说明限制并给下一步。
- 对相似度搜索结果保持克制：POI、机场/城市、酒店和网页搜索最多看前两页或前两轮；仍无匹配时，不继续机械翻页，改为说明可能不存在、名称不准或暂无库存。

## 规划内容

行程方案应尽量覆盖：
- 目标：这次出行要达成什么，哪些点是必须满足的。
- 时间窗口：日期、出发时间、返程时间、每段大致停留时长。
- 起终点：从哪里出发，最后回到哪里或到达哪里。
- 交通方式：步行、驾车、骑行、公共交通或组合方式。
- 地点清单：必去点、可选点、可跳过点。
- 顺序：按少折返、时间约束、营业状态、体力和天气排序。
- 缓冲：转场、排队、用餐、休息、突发延误的缓冲时间。
- 风险：天气、营业时间、票务、体力、路况、通勤压力、冲突日程。
- 备选：天气不好、人太多、时间不足或体力下降时怎么调整。
- 资源：多日旅行或差旅需要时，加入机票、酒店、房型和价格/库存约束。
- 下一步：需要用户确认、预订、准备、保存或安排的动作。

不同任务侧重点：
- 短外出：强调可执行时间线，突出出发、到达、关键停留和返程。
- 一日游：按上午、午餐、下午、晚上组织。
- 多日游：按 Day 1、Day 2 组织，每天保留主线、缓冲和备选。
- 多点办事：强调顺路、少折返、办事时段和失败备选。

## 工作流

1. 判断任务形态：
   - 已定时间和地点：做执行路线或时间表；需要落日历时交给 `schedule-management`。
   - 只有目的地想法：先做候选和粗计划，必要时继续在聊天里保留为待确认计划。
   - 多点外出：先定位所有地点，再排序和估算路程。
2. 收集或推断必要信息：
   - 日期/时间窗口。
   - 起点和终点。
   - 交通方式。
   - 必去点、可选点、避开的点。
   - 只能把相对日期换算成绝对日期，或从上下文里已有的用户城市、设备大概位置、日程地点等推断“候选约束”；推断结果必须标注为待确认，不要写成事实。
3. 查询事实：
   - 地点候选、坐标、周边、路线和天气交给 `location-services`。
   - 机票、机场代码、酒店和房型交给 `travel-resources`。
   - 需要实时营业、门票、活动信息时使用可用的外部搜索工具。
4. 生成方案：
   - 给出主方案和必要备选。
   - 说明每段大致顺序、交通方式、关键时间点和注意事项。
   - 多日旅行按天组织；普通外出按时间线组织。
5. 落地：
   - 用户明确要安排到日历，转 `schedule-management`。
      - 用户要继续细化，保留当前假设并迭代。

## 输出形态

- 复杂旅行或正式行程默认生成 HTML 文件：多日游、周末游、差旅、机酒组合、酒店/航班对比、多人出行、需要分享或保存的方案，都交给 `ling-report` 输出 HTML。
- 聊天内只给少量文字摘要：1-3 句话说明推荐结论、最重要风险和文件链接，不在聊天里铺完整攻略。
- 轻量外出才直接聊天输出：临时通勤、短距离接送、单点办事、一个晚餐/看展安排，且信息量很小时，用紧凑时间线即可。
- 多方案比较：列出推荐方案、备选方案和取舍理由，不堆砌所有候选。
- 短外出：用紧凑时间线。
- 一日游：按上午、午餐、下午、晚上组织。
- 多日游：按 Day N 组织，每天保留主线、缓冲和备选。
- 多点办事：按顺路顺序组织，并标出每个点失败时的替代动作。
- 用户要求攻略、报告、分享或保存成文件时，转 `ling-report` 生成 HTML；即使用户没有明确要求，只要信息量较大，也默认文件化。

回复不要展示工具名、参数或内部调用链。只给用户可执行的路线、时间、选择理由和必要提醒。

## 持久化

默认采用分层策略，复杂行程优先文件化：
- 已定档但不需要文件：用 `schedule-management` 创建或更新日程，metadata 写简短行程摘要。
- 已定档且有材料价值：生成文件后，用 `calendar_update_event(preparation=[...])` 挂到日程材料；不要把详细文件链接塞进 metadata。
- 正式攻略、报告、可分享行程、机酒组合、多日旅行或用户明确要求文件化：用 `ling-report` 生成 HTML，放在 Agent workspace 的 `reports/` 目录，回复使用标准 Markdown 文件链接。

## 关联方式

- 从计划落日程：保留用户原始目标、关键约束、推荐方案和下一步动作。
- 从计划落日程：同一趟出行默认一条跨天日程；只有用户要求拆分，或不同事项明显独立，才拆成多条。
- 从文件关联日程：正式文件作为 `preparation` 材料挂到日程；metadata 只写人类可读摘要。

## 与日程的边界

- 用户说“帮我记一下以后去”“以后有空想去”但没有日期时，不创建日程；继续在聊天里给出待确认计划或说明需要具体日期才能落日历。
- 用户给出明确日期或时间边界，并表达“提醒/安排/放日历”，交给 `schedule-management` 创建或更新日程。
- 用户要 Ling 持续留意外部变化或定期寻找新候选时，不要把未来多次检查拆成多条日程；说明当前不支持持续跟踪。
- 同一趟出行默认一条跨天日程；只有用户要求拆分，或不同事项明显独立，才拆成多条。
